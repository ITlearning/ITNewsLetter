# Discord 배치 뉴스 개수 자동 범위 설계

## 배경
- 현재 4시간마다 Discord로 뉴스를 보낼 때 워크플로우에서 `MAX_NEW_ITEMS_PER_RUN=2`로 고정되어 있다.
- 이 값이 너무 작아서 4시간 사이에 쌓인 뉴스를 충분히 보여주지 못한다.
- 하지만 단순히 10개 이상으로 늘리면 Discord 메시지 길이 제한 때문에 요약이 지나치게 잘리거나 메시지 품질이 떨어질 수 있다.

## 목표
- 기본 발송 개수를 `5~7개` 범위에서 자동으로 결정한다.
- 가능한 경우 7개까지 보내고, 메시지가 길면 6개 또는 5개로 자동 축소한다.
- 운영 기본값을 위해 GitHub Variable 등록은 요구하지 않는다.
- 필요할 때만 수동 실행 입력으로 범위를 일시 override 할 수 있다.

## 비목표
- 여러 개의 Discord 메시지로 자동 분할 전송하지 않는다.
- 우선순위 정책 자체를 전면 개편하지 않는다.
- 뉴스 본문 요약 생성 로직을 새 모델이나 별도 서비스로 교체하지 않는다.

## 접근 방식
1. 우선순위 로직으로 최대 `MAX_NEW_ITEMS_PER_RUN`개까지 후보를 선정한다.
2. Discord 본문을 조합하면서 길이 제한(`DISCORD_BATCH_MAX_CHARS`) 안에 들어가는지 검사한다.
3. 길이를 넘기면 `7 -> 6 -> 5` 순서로 뒤에서부터 항목을 줄여 재조합한다.
4. 최소 개수(`MIN_NEW_ITEMS_PER_RUN`)에 도달한 뒤에도 길이가 길면 요약을 제거하고, 그래도 길면 최종적으로 잘라낸다.

## 설정
- 코드 기본값
  - `MIN_NEW_ITEMS_PER_RUN=5`
  - `MAX_NEW_ITEMS_PER_RUN=7`
- 수동 실행 입력
  - `min_items_per_run`
  - `max_items_per_run`
- 검증 규칙
  - 둘 다 `1~15`
  - `min <= max`
  - 잘못된 입력은 실행 실패 처리

## 우선순위 정책
- 기존 `prioritize_items()` 로직은 유지한다.
- 자동 축소가 필요할 때는 우선순위가 낮은 뒤쪽 항목부터 제거한다.
- `TECH_PRIORITY_QUOTA`는 기본 `3`으로 상향한다.
- `GEEKNEWS_MAX_PER_RUN`는 기본 `1`을 유지한다.

## 관측성
- `data/last_run.json`에 아래 필드를 남긴다.
  - 설정된 `min/max`
  - 우선순위 선정 개수
  - 실제 배치 선택 개수

  - 축소 사유
  - 본문 모드(`full_summary`, `compact_summary`, `titles_only` 등)
  - 최종 본문이 잘렸는지 여부

## 검증 계획
- 스크립트 문법 검사
- 로컬에서 배치 선택 함수를 직접 호출해 아래를 확인
  - 짧은 항목: 7개 유지
  - 긴 항목: 6개 또는 5개로 자동 축소
  - 매우 긴 항목: 최소 개수에서 요약 축약 또는 잘림 처리
